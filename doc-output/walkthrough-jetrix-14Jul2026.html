Module 01

Core Objects

The four objects every piece of work flows through. A scope becomes features, each feature owns an implementation plan, and each plan generates the tasks that map to repositories and branches.

  1. Scope. A business requirement supplied by the BA.
  2. Feature. A small independently deliverable capability with its own ID and title.
  3. Implementation Plan. Holds pages, APIs, database changes, business rules, acceptance criteria, test strategy, and a loop verification checklist. Temporary until merged.
  4. Tasks. Generated automatically across Frontend, Backend, QA, and Migration. Each task owns a repository and branch mapping.

Module 02

Brownfield Workflow

How an existing project moves a feature from scope to merged DEV context. Onboarding imports the current architecture once, then every feature runs the same loop.

  1. Onboard. The project already exists. JETRIX imports pages, APIs, database, entities, relationships, and existing features. A canonical architecture context is created.
  2. Plan scope. The BA uploads scope and scope-plan breaks it into features and feature IDs, creating epics in JETRIX.
  3. Plan the feature. The developer runs feature-plan FT-1024. The plugin loads the current DEV architecture, the feature description, and existing APIs, pages, and DB, then produces the implementation plan.
  4. Review the generated plan. New login page, modified user endpoint, added MFA endpoint, DB tables for MFA secret and recovery codes, testing, acceptance criteria, security checklist, and loop validation. Stored against FT-1024. Nothing modifies canonical architecture.
  5. Implement. implement-feature FT-1024 creates branches, updates tasks, implements code, runs loop engineering, fixes failures, and generates documentation. Status becomes Development.
  6. Merge. merge-to-dev sets status to Merged to DEV and CI executes reconciliation.

Module 03

Greenfield Workflow

Starting from an empty project. After bootstrap, the workflow becomes identical to brownfield, so a team learns one loop regardless of where a project begins.

  1. Create the project. Start empty.
  2. Define repositories. Register the repos the project spans.
  3. Register environments. MAIN, DEV, STAGING, QA, PRODUCTION.
  4. Bootstrap. bootstrap-project creates an empty architecture, an empty feature catalog, and the default engineering rules.
  5. Upload initial scope. The BA uploads scope and the workflow becomes identical to brownfield.

Module 04

CI Context Reconciliation

The gate that keeps architecture honest. The pipeline compares the implementation plan against the merged code, and only a clean match promotes the DEV architecture context.

  1. Read the implementation plan.
  2. Read the merged code.
  3. Compare both and validate: expected pages created, APIs match, DB changes exist, acceptance criteria covered, tests passing, and whether any unexpected architecture is detected.
  4. On mismatch, promotion is blocked.
  5. On success, the DEV architecture context is updated automatically.
Validation checklist

Module 05

Environment Promotion

Each environment owns its own context, and promotion happens only after successful CI validation. Context follows the merge, so documentation is never updated by hand.

  1. Each environment owns its context. MAIN, DEV, STAGING, QA, PRODUCTION.
  2. Promotion is gated. It happens only after successful CI validation.
  3. Context follows the merge. Merging to an environment updates that environment's context automatically.
  4. No manual documentation updates at any stage.

Module 06

Why Feature-owned Context

Instead of branching documentation, implementation plans are attached to features. Git stays the source of code and JETRIX stays the source of architecture.

No graph branching No orphan context Natural task ownership Easy audit trail Easy rollback Git remains source of code JETRIX remains source of architecture
