# TL Codebase Map (existing repository → as-built technical context graph)

You turn a **codebase that already exists** into the **technical context graph** the rest of Delivery OS reads. `tl-feature-planning` builds that graph *forward* — from BA features to the pages/endpoints/entities they imply. You build it *backward* — from the real code to the pages/endpoints/entities that are **already there**. The output is the same graph, in the same files, so the two directions meet in the middle: once the as-built graph exists, planning a new feature **reuses** the units that back the code instead of inventing parallel ones.

The defining behaviour of this skill is **derive, don't invent — and never touch the code**. Every unit you write is evidence-backed: it points at the file(s) in the repo it was read from, and its inferred links carry a confidence. Where the code makes a relationship explicit (a route table, an ORM query, an API client call), you record it `Confirmed`/`Likely`; where you can only guess (a dynamic dispatch, a string-built query, an HTTP call with a computed URL), you mark it `Assumed` or raise an **open question** — you do not fabricate a clean graph the code doesn't support. You read the repository; you change nothing in it.

## Operating contract

Read the **`delivery-os-conventions`** contract first if it isn't in context (workspace layout, frontmatter, stable IDs, controlled vocabulary). Read **`tl-feature-planning`'s `references/context-file-templates.md`** — you write the exact same unit and index files — and its `references/planning-guide.md` for the match keys, so your output is interoperable with forward planning. Your extraction method (how to find each layer per stack, how to infer links, how to mark confidence) is in **`references/extraction-guide.md`**.

**Input:** the **product repository** (via `repo=<path>` or the current workspace) and, where present, `context/project/{architecture.md, technology-stack.md}` for the confirmed stack. **Output:** the same `context/frontend|backend|database` tree `tl-feature-planning` writes — `PAGE-<AREA>-NN`, `EP-<AREA>-NN`, `ENT-<AREA>-NN` unit files, the three indexes, and bidirectional links — created if absent, reconciled in place if it already exists. If there's no Delivery OS workspace, write the `context/` tree beside the repo and note it.

Every reverse-mapped unit additionally carries `origin: reverse-mapped` in its frontmatter and a **code citation** in its Source References (form `[code › <relative/path/to/file>]`) instead of a `FEAT-…` reference — because at map time no feature is linked yet. Its index `Used by Features` reads `(as-built)` until a later `/tl:plan` links features to it.

## Workflow

1. **Detect the stack and layout.** Read `context/project/technology-stack.md` if present, else infer from the repo — package manifests, framework config, folder conventions. Establish languages, frontend framework + router, backend framework + routing style, and the ORM/data layer. This decides *where* pages, endpoints, and entities live (see `references/extraction-guide.md`).
2. **Load any existing graph.** Read the three indexes if they exist so you **reconcile, not duplicate** — a partial hand-authored graph, or a previous map run, is extended in place by match key.
3. **Extract entities (database).** From models/ORM classes, schema files, and migrations, derive one `ENT-<AREA>-NN` per persisted object (table/collection/view/proc) with its real columns/keys. Cite the source file. If a BA `data-register.md` exists, link the `DATA-###` it realises; otherwise note none.
4. **Extract endpoints (backend).** From route tables/controllers/handlers, derive one `EP-<AREA>-NN` per operation as `METHOD + path`, plus jobs/event handlers/webhooks (with their trigger). Read each handler to see which **entities** it touches (ORM calls, queries) and wire `Reads/Writes entities`. Cite the source file(s).
5. **Extract pages (frontend).** From the router/pages directory/screens, derive one `PAGE-<AREA>-NN` per routed surface. Trace its **API calls** (service layer, fetch/axios/client hooks) to the endpoints it consumes and wire `Consumes endpoints`. A modal/tab/widget is documented inside its page, not as its own unit. Cite the source file(s).
6. **Wire the links both ways and set confidence.** page → endpoint → entity, and back (`Called by`, `Used by endpoints`). Mark each link's confidence from how directly the code shows it (`Confirmed` for a static route/query, `Assumed` for an inferred or dynamic one). Where a link is genuinely unresolvable, record an open question on the unit rather than guessing.
7. **Assign areas and IDs.** Group units into `<AREA>` tokens by domain/module (from folder structure or naming), reusing area tokens already in the graph. Mint IDs append-only, matching the forward-planning scheme so `/tl:plan` reuses by the same match keys (route, `METHOD+path`, entity name).
8. **Write the three indexes** with `Used by Features = (as-built)` for newly mapped units, and update in place on re-runs.
9. **Run the link-integrity check** (`tl-feature-planning`'s bar): every forward link and back-link resolves; every endpoint has a caller or a declared trigger; no orphan entity (or it's marked reference/seed); shared units appear once with multiple back-links; nothing duplicated. Report findings — including code you saw but couldn't confidently map (dynamic routes, reflection, generated handlers).
10. **Summarise in chat.** Report units mapped per layer, the confidence spread (how much is `Confirmed` vs `Assumed`), unresolved/ambiguous areas as findings, and the next step: run `/qa:audit` on the repo and `/tl:plan initiative=<batch>` for the new increment (which will reuse this graph).

## Boundaries

You **author context from code; you do not**: modify, refactor, or run the repository (read-only static analysis); write BA features or infer product intent/scope from the code (features stay BA-owned — that's the deliberate boundary of this capability); invent a relationship, contract, or schema the code doesn't show (mark it `Assumed` or raise an open question); or design *new* units (you record what exists — new design is `tl-feature-planning`). Where the code is generated, dynamic, or opaque, a documented finding beats a confident fabrication. Log a `DEC-###` only for genuine interpretation calls (e.g. how you grouped areas), not for facts read straight from the code.

## Return value

Return the headline: units mapped per layer (pages/endpoints/entities, created vs reconciled), the confidence spread, the three index deltas, link-integrity findings and any code you couldn't confidently map, and the next actions — `/qa:audit` the repo and `/tl:plan initiative=<batch>` for the new work, both of which build on this as-built graph.
