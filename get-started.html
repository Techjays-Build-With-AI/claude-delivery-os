Techjays Delivery OS

Get Started

An internal Claude Code plugin marketplace that standardizes how we run delivery — from raw client inputs to structured, reviewed delivery documents. Install it once, then run the agents you need.

Marketplace: techjays-delivery-os delivery-os · jetrix · ba · dev · qa · tl · doc Runs in Claude Code

Get started

Everything ships as one marketplace. You add the marketplace once (from GitHub or a local folder), install the plugins, and you're ready. Each agent lives under its own command namespace, so you get exact commands like /ba:scope, /dev:build, and /qa:audit.

The plugins at a glance

Listed in the delivery order the tabs follow: scope it (BA), build it (Dev), make it verifiable (QA), design & review it (TL), and present it (Doc).

1 Add the marketplace

Pick one way to add the marketplace. The marketplace is named techjays-delivery-os (from its marketplace.json), so every install/update command references that name regardless of how you added it.

From the published GitHub repo (tracks main)

/plugin marketplace add Techjays-Build-With-AI/claude-delivery-os

2 Install the plugins

Same for all options above. We suggest installing all of them. The agents are designed to work as one pipeline — each hands off to the next — so installing the full set gives you the complete flow and lets the Dev loop auto-plan via TL and verify against QA's gates without missing pieces. Install the core first; the order below follows the delivery flow.

/plugin install delivery-os@techjays-delivery-os
/plugin install jetrix@techjays-delivery-os
/plugin install ba@techjays-delivery-os
/plugin install dev@techjays-delivery-os
/plugin install qa@techjays-delivery-os
/plugin install tl@techjays-delivery-os
/plugin install doc@techjays-delivery-os
Recommended: install the whole suite. The plugins are lightweight and only add commands you choose to run, so there's no downside to having them all — and several depend on each other at runtime (Dev auto-plans via TL and verifies against QA's quality gates). If you'd rather start minimal, delivery-os + ba is the smallest useful set, and you can add the rest anytime.

3 Verify

Reload if prompted, then run /help. You should see the new commands:

/delivery-os:init /jetrix:init /jetrix:pull /jetrix:push /ba:scope /ba:review /ba:resolve /ba:features /dev:bootstrap /dev:build /dev:validate /dev:fix-review /dev:pr /dev:usage-eval /qa:audit /qa:plan /qa:setup /qa:health /tl:plan /tl:review /tl:resolve /tl:scaffold /tl:map /tl:maturity /doc:proposal /doc:magic-board /doc:workflow /doc:walkthrough

💡 Tip: click any command chip for a detailed breakdown — how it processes, what you get, what it covers, and a worked example.

4 Keep it up to date

After new commits are pushed (GitHub) or you edit files (local), refresh without re-adding:

/plugin marketplace update techjays-delivery-os
Switching sources (e.g. from a branch or local folder back to published main): remove first, then re-add.

5 Initialize a project workspace

Before running the BA agent, scaffold a workspace (once per project). See the Core tab for details.

/delivery-os:init my-client-project
How the review loops work (common to BA & TL & QA): a review writes an interactive HTML report. You open it, respond under each finding/question (or approve each recommendation), click Export, save the downloaded file, and run the agent's follow-up command on it (:resolve for BA/TL, :plan for QA). The agent adjudicates each answer and writes the next step. Details are on each agent's tab.

How it works

Delivery OS is a relay. Each agent does its part and hands off to the next — not by passing a file around, but by writing to a shared workspace that the next agent reads. You scaffold that workspace once with /delivery-os:init; from there the folders are the handoff. Below is who does what, who starts first, and how the baton passes — for a greenfield project (no code yet) and a brownfield one (an existing codebase).

The cast — five roles, one workspace

Business Analyst /ba

Starts every project. Turns raw client material into a living scope, then breaks it into an implementation-ready feature set.

Technical Lead /tl

Turns features into a technical context graph (pages, endpoints, entities), reviews specs, and — on greenfield — scaffolds the initial repo.

Quality Assurance /qa

Makes the repo testable: stands up the test harness and publishes the quality-gate contract the Dev loop verifies against.

Developer /dev

Builds each feature through a state-driven loop, validates it against the acceptance criteria and QA's gates, and prepares a PR for human review.

Documentation /doc

Runs alongside — produces client-facing artifacts (proposal, workflow walkthrough, magic board) from the BA scope, mainly during discovery/sales.

The golden rule: an agent only reads another agent's published output, never its working notes. The BA writes ba-output/ and context/features/; the TL writes the context/frontend|backend|database graph; QA writes qa-output/quality-gates.md; Dev writes each feature's dev/ folder and the PR summary. Because it's all in the workspace (and the repo), any agent — or teammate — can pick up where the last left off.
Centralizing to Jetrix? The same relay runs, but the workspace becomes a working copy of a central Jetrix project that is the source of truth. Bind once with /jetrix:init, run /jetrix:pull before you work, and /jetrix:push to publish each agent's output back as structured records. See the Jetrix tab.

A Greenfield — a brand-new project with no code yet

Nobody has written a line of application code. The job is to go from client material all the way to built, validated features. The order matters because each step needs the previous one's output.

  1. Set up the workspace (once). Anyone runs /delivery-os:init my-project — this creates the single container folder every agent reads and writes within.
  2. BA scopes the work. The Business Analyst runs /ba:scope add "…PRDs, FRDs, meeting notes…" to build the living scope.md, then tightens it with the /ba:review → /ba:resolve loop. (The Doc agent can draft the client proposal from this scope in parallel.)
  3. BA breaks scope into features. /ba:features initiative=<name> decomposes the scope into one folder per feature under context/features/, each with acceptance criteria, dependencies, and open questions — and stamps them all with the initiative name. This is the handoff to the build team.
  4. TL stands up the repo. Because there's no codebase and the stack is an architecture decision, the Technical Lead runs /tl:scaffold (or the Dev agent routes there automatically via /dev:bootstrap). It asks for any undecided stack choice with a recommendation, then produces a real repo with a green base build.
  5. QA makes it testable. On the fresh repo, QA runs /qa:audit → /qa:plan → /qa:setup to stand up the test harness (frameworks, CI, coverage, e2e) and publish qa-output/quality-gates.md — the contract that defines what "verified" means here.
  6. TL plans the features. /tl:plan initiative=<name> maps each feature to its pages, endpoints, and entities in the technical context graph. (You can skip running this yourself — the Dev loop auto-plans a feature via the TL if it isn't planned.)
  7. Dev builds each feature. /dev:build initiative=<name> picks up a ready feature, implements it in an isolated branch, validates it against the acceptance criteria and QA's gates, repairs actionable failures, and stops at HUMAN_REVIEW with a PR summary. It never merges.
  8. A human reviews and merges. Then Dev moves to the next feature. Repeat until the initiative is done.
GREENFIELD — the relay

  init   /delivery-os:init                       →  workspace created
   │
  BA     /ba:scope → /ba:review → /ba:resolve     →  ba-output/scope.md   ──▶ (Doc: /doc:proposal)
  BA     /ba:features initiative=payments-v2      →  context/features/*
   │
  TL     /tl:scaffold   (or /dev:bootstrap)       →  real repo + green base build
  QA     /qa:audit → /qa:plan → /qa:setup         →  qa-output/quality-gates.md
  TL     /tl:plan initiative=payments-v2          →  context/frontend|backend|database
   │
  DEV    /dev:build initiative=payments-v2        →  feature branch + PR at HUMAN_REVIEW
   │
  YOU    review & merge                           →  repeat per feature
You don't have to memorize the order. /dev:build has three gates that route you: no repo → bootstrap (TL scaffold); no test harness → QA setup; feature not planned → TL auto-plan. If something upstream is missing, it tells you exactly which command to run.

B Brownfield — an existing codebase you're adding to

The app already exists and (often) some scope with it. You're adding an increment, usually with several developers working in parallel. Two steps from greenfield drop away — you don't scaffold a repo, and you don't re-scope what's already built — and the initiative becomes the tool that keeps each developer focused on their own batch.

  1. BA scopes only the increment. Run /ba:scope add "…just the new material…" to update the living scope, or point /ba:features straight at an existing scope document. Keeping the scope to the new work is the cleanest way to avoid regenerating features you've already shipped.
  2. BA breaks it down under a new initiative. /ba:features initiative=<new-batch> creates folders for just the new features and tags them. If your scope covers the whole system, mark already-shipped features Released in the index so only the new ones flow downstream. Each developer uses their own initiative name, so the shared context/features/ stays sortable.
  3. TL maps the existing code into the context graph. Run /tl:map repo=<path> once to reverse-engineer the repo — deriving the frontend pages, backend endpoints, and database entities and their relationships that are already there, as an as-built graph. This is the brownfield equivalent of greenfield's scaffold step: it gives the build team the technical context the code already embodies, so planning the new increment reuses it.
  4. QA checks the existing setup. /qa:audit on the real repo shows what testing already exists and where the gaps are; /qa:setup fills only what's missing and updates the quality-gate contract. (No need to start from scratch if a harness is already there.)
  5. TL plans just the new batch. /tl:plan initiative=<new-batch> plans only that developer's features, ignoring everyone else's in-flight work.
  6. Dev builds into the existing repo. /dev:build initiative=<new-batch> repo=D:\path\to\app — the repository gate sees a real repo and treats it as brownfield, so there's no scaffolding step. It builds, validates against the gates, and hands off a PR.
  7. Human reviews and merges, same as greenfield.
BROWNFIELD — what changes

  BA     /ba:scope add "just the new increment"        →  updates ba-output/scope.md
  BA     /ba:features initiative=reporting-v2          →  new feature folders only
         (mark already-shipped features Released)
  TL     /tl:map repo=                           →  as-built context graph from the code
  QA     /qa:audit → /qa:setup   (on the existing repo) →  fills test-setup gaps
  TL     /tl:plan  initiative=reporting-v2             →  plans this batch (reuses mapped units)
  DEV    /dev:build initiative=reporting-v2 repo= →  builds into the existing repo
         (no /tl:scaffold — the repo already exists; /tl:map replaces it)
Multiple developers, one repo. Because context/features/ lives in the repo, everyone's in-flight features share it. The initiative is what keeps a developer focused on their own batch: /ba:features stamps it, and /tl:plan and /dev:build filter by it, so Dev B never trips over Dev A's four unfinished features. Pair it with per-initiative branches for clean merges.

How the baton passes — every handoff

Each arrow in the relay is one agent's published output becoming the next agent's input:

From → ToHandoff surface (in the workspace)
BA → TL, QA, Devcontext/features/ — one folder per feature (feature, acceptance criteria, dependencies, open questions) + the feature-index
BA → Docba-output/scope.md — the living scope the proposal/workflow docs draft from
TL → Devcontext/frontend|backend|database — the pages/endpoints/entities graph the feature maps to
TL → Dev (greenfield)The scaffolded repo + context/project/{architecture, technology-stack, coding-standards}.md
Code → TL graph (brownfield)/tl:map reads the existing repo → the same context/frontend|backend|database units, marked as-built, that /tl:plan then reuses
QA → Devqa-output/quality-gates.md — the checks, commands, and thresholds that define "verified"
Dev → Humandev/pr-summary.md + the feature branch at HUMAN_REVIEW
Everyone → Everyoneshared-context/ (actors, glossary, decisions) and the DEC-### decision log
The short version: BA answers what, TL answers which pages/APIs/entities, QA answers what "verified" means, Dev answers the working, validated code, and Doc presents it to the client. Each consumes the previous one's output and never re-does it. Greenfield adds a one-time TL scaffold + QA setup; brownfield skips the scaffold and leans on the initiative to keep parallel work separate.

Core — delivery-os

The shared foundation every other plugin depends on. It scaffolds the project workspace, defines the cross-agent document contract (frontmatter, stable IDs, shared vocabulary), and ships the canonical Techjays templates. Install this first.

Commands

/delivery-os:init <project-name>

What it does

Creates one named container folder for the project so everything Delivery OS owns is visible in a single place (and a single folder to delete for a clean slate). It does not pre-create a rigid raw-files taxonomy — sources are added later via /ba:scope.

Run it

/delivery-os:init acme-invoice-processing

It scaffolds:

acme-invoice-processing/
├── README.md            # workspace manifest
├── intake.index.md      # living source registry (maintained by /ba:scope)
├── artifacts/           # generated source summaries (categories created on demand)
├── shared-context/      # project profile, glossary, stakeholders, systems, decisions
├── ba-output/           # BA deliverables: scope.md, registers, scope-reviews/
│   └── intake-runs/
└── final/               # approved, client-facing deliverables
Why it matters: all agents read and write only within this contract, so the BA's output is reliably consumed by the Dev, QA, TL (and Doc) agents. Downstream agents create their own output folders (dev-output/, qa-output/, tl-output/, …) on first use.
Tip: if you give no name, the command asks for one rather than scaffolding into your current directory.

Jetrix — jetrix

Optional. Use Jetrix when a project's context should live in one central store that many repos and teammates share, rather than only in a local workspace. When a project is on Jetrix, Jetrix is the single source of truth for all project context — glossary, scope, registers, features, and the technical context graph — and your local .jetrix/ folder is a disposable working copy, exactly like a git checkout.

Local-only vs Jetrix mode. A purely local project uses /delivery-os:init and never touches Jetrix — skip this tab. A centralized project additionally binds to Jetrix with /jetrix:init and syncs with /jetrix:pull and /jetrix:push.

Commands

/jetrix:init <project-slug> /jetrix:pull /jetrix:push

The working-copy model

You pull the slices you need into a local cache, do a whole piece of work locally (fast, reviewable, no round-trip per read, works offline), and when you're satisfied you push it back. A push is a deliberate commit, not an autosave — nothing local is published until you push.

  1. Bind once. /jetrix:init <project-slug> records which Jetrix project this workspace maps to in the committed .jetrix/project.json and scaffolds an empty, gitignored .jetrix/cache/. It fetches no content — binding stays fast.
  2. Pull. /jetrix:pull fills the cache from Jetrix. It is incremental — a per-section hash means it re-downloads only what changed, so pulling often is cheap and staleness is a non-issue. Full pull (no args) grabs project + apps + all content; scoped flags (--apps, --scope, …) refresh one section.
  3. Work locally. The BA, TL, and Dev agents read and write the cache like any workspace.
  4. Push. /jetrix:push publishes your changes back as structured records (requirements, workflows, features, endpoints, …), not a document blob. It is idempotent (upsert by stable id, so re-pushing updates instead of duplicating), transactional (propose → preview a diff → commit), and safe (pull-before-push; on a both-sides-changed conflict it stops rather than clobber a Jetrix edit).

Two files, opposite lifecycles

PathTracked?Holds
.jetrix/project.jsonCommittedIdentity + wiring only: project id, environments, and each app (repo) with its environment→branch map. No content, no secrets.
.jetrix/cache/GitignoredThe disposable working copy of the Jetrix store, plus the manifest (per-section hashes + id-map). Safe to delete anytime; a pull rebuilds it.

Greenfield: creating apps

An app is a repo (frontend, backend, mobile, a service). At greenfield there are none yet, so project.json starts with an empty apps list. When the technical layer needs apps, you create them in Jetrix — each with its repo link and its environment→branch mapping (dev / staging / uat / prod) — and then /jetrix:pull --apps brings them into project.json. The logical implementation needs (which pages, endpoints, and entities a feature requires) are app-agnostic and can be produced before apps exist; only the app-binding waits on the pull.

JETRIX — the sync loop

  BIND   /jetrix:init acme-portal      →  .jetrix/project.json (identity) + empty cache
  PULL   /jetrix:pull                  →  cache filled from Jetrix (incremental)
   │
  WORK   /ba:scope, /ba:features, …    →  edits in the local working copy
   │
  PUSH   /jetrix:push                  →  structured records upserted to Jetrix (propose → diff → commit)
   ·
  (apps) create apps in Jetrix  →  /jetrix:pull --apps  →  repos + env→branch wiring land in project.json

BA Agent — /ba

The Business Analyst has two jobs: during discovery it's the project's author & memory (processes client artifacts into a living scope and registers); once a scope exists it becomes its paranoid critic (breaks the scope down feature by feature, bounds each one, validates it against the client's examples, and scores how estimate-ready it is).

Commands

/ba:scope /ba:review /ba:resolve /ba:features

Build the living scope — /ba:scope

Point it at your client material — folders or Google Drive. It references originals in place (never copies or moves them), maps everything into summaries under artifacts/, and writes the module-centric scope document (ba-output/scope.md) in the Techjays template. Run it again whenever new material arrives — it updates the same living document.

# build the scope from the first batch of material
/ba:scope add "transcripts in D:\acme\meetings, requirements at <drive-link>, invoice archive in D:\acme\invoices for reference only"

# later — fold in new notes or a client-answers doc
/ba:scope add "answers in D:\acme\client-answers.md"
/ba:scope mode=incremental

Modes: auto (default) · incremental (only new/changed) · full-refresh · dry-run (report, write nothing) · index-only · classify-only.

Raises the open questions

Every run regenerates ba-output/client-questions.md — a clean, prioritized list you hand straight to the client, with space for answers.

Folds answers back in

Point /ba:scope at the filled-in answers (or new meeting notes); it matches each to its question, closes it, and updates the scope.

The living-scope loop: /ba:scope to build → hand client-questions.md to the client → /ba:scope add "answers…" to fold responses in and close questions → repeat. One scope document that keeps getting sharper.

Scope review — /ba:review

Reviews the scope like a paranoid BA before it goes out for estimate. Defaults to ba-output/scope.md; you can also point it at any scope file.

# review the living scope in the workspace
/ba:review

# or review a specific document
/ba:review docs/acme-scope.docx

What the review does

Bounds every feature

Enumerates the categories/buckets, the definitions used to classify them, what's covered vs excluded, and mandatory vs optional fields.

Scores coverage /10

Judges nine business dimensions (in/out scope, business rules, info & data, exceptions, acceptance, …) per feature.

Validates examples

Checks each feature against the client's shared examples: Pass / Partial / Conflict / No-examples.

Business, not technical

Judges what the business wants; leaves schema, APIs, and system design to the TL review.

Boundedness caps the score: a feature whose function is clear but whose boundary is undefined ("classify the invoice" with no defined types) is capped at Weak until the buckets, classification logic, and exclusions are pinned down. Each feature is tagged Bounded / Partially-bounded / Unbounded.

Output lands in ba-output/scope-reviews/scope-review-<timestamp>.{html, md, json} — open the .html as an interactive dashboard.

Respond & close — /ba:resolve

The review raises scope questions; this loop closes them and folds the answers back into the scope.

  1. Open scope-review-<timestamp>.html in a browser and go to Open questions.
  2. Type an answer under each and pick an intent (resolve / accept-assumption / need-info / wont-fix).
  3. Click Export responses → save the downloaded scope-review-<timestamp>-responses.md into ba-output/scope-reviews/.
  4. Run resolve on it:
/ba:resolve ba-output/scope-reviews/scope-review-<timestamp>-responses.md

The agent adjudicates each answer, gives the exact scope edit, promotes confirmed answers into the scope and registers (decisions, assumptions, clarifications), recomputes the feature scores and verdict, and writes a new round. Repeat until no questions remain.

Verdicts: Scope-ready → Estimate with caveats → Significant gaps (clarify before estimate) → Not scope-ready. Any unresolved Blocker caps the verdict — resolving it lifts the cap.

Dev Agent — /dev

The Developer agent takes an approved, TL-planned feature and builds it — through a controlled, state-driven loop rather than a one-shot code dump. It reads the BA feature context and the TL technical graph, validates readiness, plans, implements in an isolated branch, validates against the acceptance criteria, repairs actionable failures within retry limits, tracks state, and prepares a pull-request handoff. It escalates ambiguity, risk, and scope decisions instead of guessing, and it never merges or deploys.

Commands

/dev:bootstrap /dev:build /dev:validate /dev:fix-review /dev:pr /dev:usage-eval
Needs the BA breakdown; plans TL for you; verifies against QA's gates. The dev agent consumes the BA feature breakdown (/ba:features), so that must exist. TL planning is automatic: /dev:build opens with a planning gate that checks whether the feature is already split into pages/endpoints/entities and, if not, auto-plans it via tl-feature-planning before building. Its test-harness gate reads QA's qa-output/quality-gates.md to know how to verify — and routes to /qa:audit → /qa:setup if no harness exists yet (see the QA tab).

Greenfield? Bootstrap once — /dev:bootstrap

The dev loop builds into a repository. On a brand-new (design-only) workspace there isn't one — and the agent won't scaffold on a guessed stack. So /dev:build has a repository gate: on project-zero it routes to bootstrap instead of dead-ending. Run this once at the start of a greenfield project.

# stand up the initial repo (delegates to the TL scaffold)
/dev:bootstrap

# already have a repo? point at it — bootstrap isn't needed
/dev:build FEAT-USR-02 repo=D:\acme\supplier-platform
How it works: bootstrap delegates to the TL's tl-project-scaffold — it reads the architecture, asks you (with a recommendation) for any missing stack decision, scaffolds the skeleton + tooling + coding-standards.md + a green base build, and logs each choice as a DEC-###. If the tl plugin isn't installed it blocks and asks you to run /tl:scaffold — it never guesses the stack. Once done, /dev:build proceeds normally.

Build a feature — /dev:build

Pass a feature id/folder, or leave it blank to pick the next feature at READY_FOR_DEV. It runs the full loop and stops at HUMAN_REVIEW (or BLOCKED).

# build one feature end to end
/dev:build FEAT-SUP-001

# or let it pick the next ready feature, against a specific repo
/dev:build repo=D:\acme\supplier-platform

The loop it runs

Select feature (or next READY_FOR_DEV) → acquire lock
→ Read BA + TL context, summarise → Validate readiness (gate)
→ Analyse code-level impact → Write the technical plan
→ Create isolated branch/worktree → Implement (scoped, with tests)
→ Validate suite → Map results to acceptance criteria
→ Repair actionable failures (bounded) → Self-review + security pass
→ Update delivery status + feature tracker → Prepare PR handoff
→ Hand off for HUMAN_REVIEW   (never merges or deploys)
It reports as it goes. The loop broadcasts each state transition in chat (e.g. IN_DEVELOPMENT → TESTING) with a short heartbeat on long phases, and surfaces a blocker as an explicit inline error block — so you can watch where it is without opening any markdown.

Real execution

Creates the feature/FEAT-… branch, edits code, and runs lint/type/test/build in the shell against the real repo.

Acceptance = done

Completion is a filled acceptance map — every criterion tied to a validation method and an evidence artifact. A green unit run alone is not enough.

Bounded repair loop

3 focused fixes per failure, 2 broad validation cycles, 2 plans per feature — then it escalates instead of retrying blindly.

Escalates, never guesses

Ambiguous rules, schema/data-loss risk, unclear authz, unavailable dependencies, and security concerns become a structured, decision-ready note.

The state model

Each feature moves through explicit states — the fine-grained state lives in dev/delivery-status.md and is mirrored into the BA feature-index.md/status.md:

BACKLOG → READY_FOR_DEV → IN_PLANNING → BLOCKED → IN_DEVELOPMENT
        → TESTING → REVIEW_FIXES → READY_FOR_PR → HUMAN_REVIEW
        → APPROVED → MERGED → RELEASED
It stops at HUMAN_REVIEW. The agent owns every transition up to the handoff; APPROVED / MERGED / RELEASED are human- or deploy-owned and it only records them. It never merges, deploys, deletes production data, modifies secrets, disables a security control, or expands scope without approval.

Validate only — /dev:validate

Run the checks (lint, type, unit, integration, API-contract, build, migration, security, e2e) and map each result to the feature's acceptance criteria — no code changes. When QA's quality-gate contract is present, it runs the Required gates that contract defines.

/dev:validate FEAT-SUP-001

Reports the acceptance pass/fail table and flags each failure as actionable (fix via /dev:build) or blocking (escalate). Output lands in context/features/<slug>/dev/acceptance-map.md.

Fold in review feedback — /dev:fix-review

Re-enters the loop at the review-fixes step with the reviewer's comments as the failure set, addresses actionable ones within the retry limits, re-validates, and refreshes the PR summary. Comments needing a product/architecture/security decision are escalated.

/dev:fix-review FEAT-SUP-001 feedback=D:\acme\pr-42-comments.md

Prepare the PR handoff — /dev:pr

For a feature that's implemented and validated: verifies the completion criteria, then writes a review-ready summary (purpose, scope, technical approach, affected pages/APIs/services, tests run, acceptance status, risks, follow-ups, reviewer instructions) and moves the feature to HUMAN_REVIEW.

/dev:pr FEAT-SUP-001
Completion gate: if a mandatory criterion isn't met (acceptance criteria not validated, tests/build failing, an unresolved blocker), it won't write a hopeful summary — it reports exactly what's missing and points you back to /dev:build.

What it writes (per feature)

Alongside the BA's files, under a dev/ subfolder so nothing collides — plus a cross-feature dev-output/feature-tracker.md:

dev/decisions.mdTechnical decisions (also appended as DEC-### to the shared decision-log)
dev/pr-summary.mdThe review-ready PR handoff
dev/escalation-<n>.mdA structured blocker note when the feature goes BLOCKED
How it fits Delivery OS: the dev agent is a consumer — it reads the BA feature folders, the TL context/frontend|backend|database graph, shared-context/, the BA registers, and QA's quality-gates.md, and never re-runs BA discovery or TL planning. It writes produced_by: dev files and mirrors feature state into the BA index using the shared vocabulary. Full guide: plugins/dev/dev_readme.md.

Usage & efficiency evaluation — /dev:usage-eval

A separate, opt-in capability bundled with the dev plugin — independent of the delivery loop above. Instead of building a feature, it evaluates how efficiently you (or a teammate) drive Claude Code and generates an interactive report card. It has two parts: silent telemetry hooks that quietly accumulate per-session activity in the background, and the agentic evaluator you invoke on demand.

# evaluate every recorded session and open the dashboard
/dev:usage-eval

# scope it: recent sessions only, a single session, or don't auto-open the browser
/dev:usage-eval since=2026-07-01 limit=20
/dev:usage-eval session=<session-id> no-open

How it works

Hooks (silent, always on)            Evaluator (/dev:usage-eval)
  UserPromptSubmit ┐                   Read ~/.claude/metrics/*.json
  PreToolUse       ├─►  ~/.claude/     → Infer persona per session
  Stop             ┘    metrics/*.json → Score upfront context + friction
                                       → Adaptive 0–100 efficiency score
                                       → Build & open ~/claude-usage-report.html

Silent, non-blocking

The hooks append normalized events to ~/.claude/metrics/<session>.json and fail silently — they never slow or block a session. Only high-signal metadata is kept (event type, tool names, file paths, prompts); code bodies and secrets are not stored.

Persona-aware scoring

Infers each session's role (Developer, BA, TL, QA, Sales/BD, PM) and grades relative to that persona's best practices — precise upfront context and tight turn counts score well; vague openers that force re-prompts don't.

Coaching, not surveillance

Every strength and improvement is tied to something visible in the logs. The read is supportive and specific — good iteration on a hard problem isn't penalized; only avoidable, under-specified rework is.

What it generates

A standalone dark-themed ~/claude-usage-report.html — efficiency score, primary persona, key strengths, improvement areas, a grade-distribution chart, and per-session highlights with letter grades.

Telemetry starts at install. The hooks only record on machines where the dev plugin is installed and node is on the PATH, so the report reflects sessions after install — not historical usage. Run a few normal Claude Code sessions first, then /dev:usage-eval. Click the /dev:usage-eval pill above for the full breakdown.

QA Agent — /qa

The Quality Assurance agent makes a repository properly testable and defines, in a machine-readable contract, exactly what "verified" means for it — so the Dev delivery loop validates features against a real, agreed bar instead of whatever tooling happens to exist. It audits the current test setup, recommends what's missing for you to approve, stands up the harness, proves it green, and publishes the quality-gate contract the Dev loop reads.

Commands

/qa:audit /qa:plan /qa:setup /qa:health
The one boundary that defines it: QA owns the test harness and strategy — frameworks, config, CI wiring, coverage thresholds, the e2e harness (e.g. Playwright config + a page-object/fixtures skeleton), test-data factories, mocking, conventions, and the quality-gate contract. The Dev agent writes the per-feature tests inside that harness, and /dev:validate runs them. QA builds the kitchen; Dev cooks each dish. QA never writes a feature's tests and never weakens a check to go green.

Audit the setup — /qa:audit

Read-only. Point it at a repo (or use the current workspace). It detects the stack and scores the test setup across 12 areas — unit, integration, e2e, coverage, lint, format, type-check, CI, fixtures, mocking, contract testing, conventions — recording every gap as a stable QAF-### finding with a severity and a recommended fix. It changes nothing in the repo.

# assess an existing repo
/qa:audit repo=D:\acme\supplier-platform

# or the current workspace
/qa:audit

Scores 12 areas /10

Verdict bands: Not testable → Major gaps → Workable → Solid. Any Blocker caps it.

A decision surface

Each QAF-### recommendation carries a recommended option; you approve, skip, or defer it — QA acts only on what you approve.

Output lands in qa-output/test-audit-<timestamp>.{html, md, json} — open the .html for the interactive report. Set Adopt / Skip / Defer per recommendation, then Export approvals.

Plan the setup — /qa:plan

Feed it the approvals file exported from the report. It reloads the audit, plans only the recommendations you approved, and drafts the quality-gate contract.

/qa:plan qa-output/test-audit-<timestamp>-approvals.md

Writes qa-output/test-setup-plan.md (ordered steps, tools, CI changes, thresholds) plus a draft qa-output/quality-gates.md.

Build the harness — /qa:setup

Implements the plan in an isolated branch: installs and configures frameworks, wires CI, adds fixtures/mocks and coverage thresholds, scaffolds the e2e harness, writes the testing conventions — and proves a green smoke run. Then it finalizes the quality-gate contract.

/qa:setup
Never green-by-weakening. QA may not disable, skip, delete, or loosen a check or threshold to pass — that's an escalation with the trade-off named. Retry limits mirror the Dev loop (3 focused fixes per failing step, 2 broad smoke re-runs, 0 strategy guesses), and it never merges or deploys — the setup branch is handed off for human review.

Keep it honest — /qa:health

Re-runs the Required gates against the current repo to catch drift — a disabled check, a dropped coverage floor, a broken e2e config — and reports the deltas. It won't "repair" by lowering a gate; a threshold change is a human DEC-###.

/qa:health

The flow

/qa:audit   → scored gap report (HTML + MD + JSON)
   ↓  you review, set Adopt/Skip/Defer, export approvals
/qa:plan    → test-setup-plan.md + draft quality-gates.md
   ↓  you review the plan
/qa:setup   → build the harness in a branch → prove a green smoke run
   ↓            → finalize quality-gates.md (harness_status: Active)
/qa:health  → re-check the live harness → report drift
How it plugs into the Dev loop: qa-output/quality-gates.md is the contract. The Dev readiness gate reads it to confirm a usable harness exists (and routes to /qa:audit → /qa:setup if not, the same way project-zero routes to /dev:bootstrap); /dev:validate reads it to know which checks are Required, the exact command for each, and the bar (coverage floor, when e2e/contract tests are mandatory). So the Dev loop stops guessing the tooling and verifies against a standard you set. Full guide: plugins/qa/qa_readme.md.

TL Agent — /tl

The Technical Lead reviews a technical specification for an applied-AI system the way a seasoned TL would before a team commits to building it. It reads the whole document, scores each build-readiness area out of 10 with concrete findings and fixes, and closes findings through a resolution loop. It also plans features into a technical context graph and scaffolds the initial project repo.

Commands

/tl:plan /tl:review /tl:resolve /tl:scaffold /tl:map /tl:maturity

Review a spec — /tl:review

Point it at any spec / architecture / HLD / SRS document (.md, .docx, or .pdf). A workspace is optional — it works standalone.

/tl:review docs/support-copilot-spec.md

What it scores (out of 10 each)

GroupAreas
Architecture & DesignOverview & problem framing · Architecture & system design · Feature flows & system sequencing
Engineering ContractsData schema & modeling · API contracts · Libraries & tech stack
Applied AI / LLM (if an LLM is involved)Observability & traceability · Evaluation strategy · Feedback loops · AI compliance & safety
Delivery & OperationsInfrastructure setup · CI/CD & release management · Cost projection (infra + LLM/API + third-party)
Verdict: the overall score (average of applicable areas) maps to Ready to build → Build with caveats → Significant gaps → Not ready. Non-AI systems mark the Applied-AI areas N/A. Any unresolved Blocker caps the verdict. Findings carry a severity (Blocker / Major / Minor / Nit) and a stable FND-### ID.

Output lands in tl-output/spec-review-<timestamp>.{html, md, json} — open the .html for the interactive report.

Respond & close — /tl:resolve

  1. Open spec-review-<timestamp>.html and go to Open items.
  2. Type a response under each finding and pick an intent (resolve / accept-risk / need-info / wont-fix).
  3. Click Export responses → save the downloaded spec-review-<timestamp>-responses.md into tl-output/.
  4. Run resolve on it:
/tl:resolve tl-output/spec-review-<timestamp>-responses.md

The agent adjudicates each response (Resolved / Accepted-risk / Needs-verification / Won't-fix), recomputes the score and verdict, logs decisions, and writes a new round. A resolved Blocker lifts the verdict cap.

Standalone or workspace: with a Delivery OS workspace it also reads ba-output/scope.md and shared-context/ for grounding; without one it writes the report beside the reviewed document.

Scaffold the project — /tl:scaffold

On a greenfield project there's no codebase yet — and the dev agent won't scaffold on a guessed stack, because the stack is an architecture decision. /tl:scaffold is where that decision is made and executed. It reads the tech spec / architecture, and for any required stack choice the architecture doesn't pin down it asks you with a recommendation — then stands up the initial repo.

# scaffold from a spec, into a chosen repo path
/tl:scaffold docs/architecture.md repo=D:\acme\supplier-platform

# or read context/project + shared-context, ask where needed
/tl:scaffold

Executes a confirmed stack

Follows what the architecture pins down; recommends-and-asks for the rest (framework, DB, hosting, layout). Never silently guesses a critical choice.

Green base build

Skeleton + lint/format/type/test/build tooling + one passing test, verified install → lint → type → test → build — the exact readiness item dev checks.

Output: a real, git-initialised repository plus context/project/{technology-stack, architecture, coding-standards}.md, with every stack choice logged as a DEC-###. Then /dev:build <feature> can run.

Onboard an existing codebase — /tl:map

The brownfield counterpart to /tl:plan. Where planning builds the context graph forward from features, /tl:map builds it backward from the real code: it reads an existing repository and derives the frontend pages, backend endpoints, and database entities — and their relationships — that are already there. It writes the same context/frontend|backend|database unit files, so once the as-built graph exists a later /tl:plan reuses those units instead of duplicating them.

# map an existing repo into the context graph
/tl:map repo=D:\acme\legacy-portal

# map just one layer, or a slice of a monorepo
/tl:map repo=D:\acme\legacy-portal layers=backend,database scope=services/billing

Derived from code, not invented

Pages from the router, endpoints from route handlers (with METHOD + path), entities from models/migrations — each unit cites the source file it was read from.

Honest confidence

Every inferred link is marked Confirmed (static route/query) or Assumed (dynamic/opaque); genuinely unresolvable links become open questions, not fabrications.

Read-only on the code. It's static analysis — it never modifies, refactors, or runs the repo, and it never authors BA features or product scope from the code (that stays the BA's). Units are stamped origin: reverse-mapped; when you later run /tl:plan, features link into these same units.
Brownfield onboarding, in order: /ba:scope (scope from your existing docs) → /ba:features initiative=<batch> → /tl:map repo=<path> (build the as-built graph once) → /qa:audit → /tl:plan initiative=<batch> (reuses the mapped units) → /dev:build initiative=<batch> repo=<path>.

Score production maturity — /tl:maturity

A read-only maturity audit of a built project. Where /tl:review scores a design document and /qa:audit scores the test harness, /tl:maturity scores the running system's reality across four domains — Code Quality & Maintainability, Test Quality & Verifiability, Infrastructure & Operations, and Security — and answers "is this production-grade, and where's the risk?" It prefers the project's own enforced tooling as evidence (lint, coverage, SARIF), tags every finding evidenced or attested, and reports a maturity score plus a separate Audit Confidence so a high score built on little measurement can't hide.

# score the current workspace / repo
/tl:maturity

# point at a repo; require the QA baseline before scoring
/tl:maturity repo=D:\acme\orders-service strict-baseline
Reminds you to run /qa:audit first. In preflight it checks whether the mandatory QA baseline (enforced lint, coverage gate, dependency + secret scanning) is in place. If it isn't, it points you at /qa:audit → /qa:setup and, by default, still scores in degraded mode at reduced Audit Confidence (with strict-baseline it stops instead). It is read-only on the repo — it installs nothing, changes no code or tests, never fabricates a score it couldn't measure, and routes baseline gaps back to QA rather than fixing them itself.
Where it fits the three audits: /tl:review scores the spec (before build) · /qa:audit scores the test harness (loop-readiness) · /tl:maturity scores the built system (production-readiness), consuming both.

Doc Agent — /doc

The Documentation agent standardizes how the org produces client-facing artifacts, so every document looks and reads like it came from one company. Phase 1 ships the client proposal: a single-file, print-to-PDF HTML deck built to the Techjays proposal standard and style system.

Commands

/doc:proposal/doc:magic-board/doc:workflow/doc:walkthrough

Generate a proposal — /doc:proposal

Pass the client name and their domain. The agent pulls the client's logo from the domain, samples their brand accent, and (when a Delivery OS workspace exists) auto-drafts the problem, solution, and value from ba-output/scope.md using the client's real numbers.

# minimal — logo + accent from the domain, content from the scope if present
/doc:proposal client="Harry Grodsky & Co." domain=grodsky.com

# with a title and custom rules
/doc:proposal client="Acme" domain=acme.com title="AI-Powered Estimating" "lean 7-section form; add a data-security section; Phase 1 fixed at $50k"

Branded from the domain

Client logo (via logo.clearbit.com, favicon fallback) on the title pair + closing, and their brand color as the accent.

Auto-drafted from the scope

Problem, solution workflow, and value numbers pulled from the BA scope. Never fabricates metrics — unknowns become [[NEEDS: …]] placeholders.

Print-to-PDF, always

Breakpoints scoped to screen, @page rules, dark→white invert, and a Print / Save PDF button. Save-as-PDF just works.

Your custom rules

Add sections, set tone, fix pricing, pick the lean form — passed in free text and layered on the standard.

What you pass

  • client="<Name>" — required.
  • domain=<client.com> — strongly recommended (drives the logo and accent color).
  • title="…" — optional; drafted from the scope if omitted.
  • Any free text — treated as custom rules and layered on the standard.
Export to PDF: the deck writes to doc-output/proposal-<client>-<timestamp>.html. Open it in Chrome/Edge, choose Print → Save as PDF, and enable "Background graphics" so the accent borders and fills render. For an offline-portable file to email or archive, ask the agent to base64-embed the client logo.
Voice rules are enforced: no em-dashes, no "not X but Y" phrasing, the client is addressed by name, and the Value page carries the client's real numbers. Full guide: plugins/doc/doc_readme.md.

Present a workflow — /doc:magic-board

A magic board is a self-contained, Miro-style infinite-canvas HTML file: cards laid out in space with pan (drag), zoom (scroll), and a guided fly-through tour that frames each card full-screen as you click next / next / next. The best way to show a workflow or your understanding of a system to a client or team.

# a workflow board, drafted from the BA scope when present
/doc:magic-board topic="Acme invoice-processing workflow" kind=workflow

# a system deep-dive with custom emphasis
/doc:magic-board topic="Our understanding of the claims system" kind=system "feature the fraud-check subsystem; ~10 cards"
  • topic="…" — what the board is about (required or inferred from the scope).
  • kind=… — the arc: workflow (default) · journey · system · concept · strategy · roadmap.
  • Free text — extra source material or rules (which systems to feature, card count, brand accent).
Presenting: open doc-output/board-<topic>-<timestamp>.html in any browser. Walk the tour with the on-screen next/prev or arrow keys, drag to pan, scroll to zoom, and press O for the overview. It's one self-contained file that opens offline.

Walk through the scope — /doc:workflow

An interactive workflow document: a Techjays-standard HTML page with a fixed sidebar and a phased SVG swimlane (phases as rows, actors as columns) with hover-over tooltips on every step. Each phase gets a before/after value panel (current pain → AI solution + hours saved) and there are KPI cards and a systems retained/retired grid.

# drafts phases, actors, and value panels from the BA scope when present
/doc:workflow project="Acme Corp — Employee Onboarding"

# with source material / emphasis
/doc:workflow project="Claims Intake" client="Acme" "actors: Agent, AI Platform, Adjuster; feature the fraud-check exception"

Phased swimlane, hover detail

Vertical swimlane of the scope; hover any step for its bullets, hover a KPI for the math.

Before / after + efficiency

Per-phase value panel: current pain → AI solution, with the hours saved called out.

Viewing: open doc-output/[CLIENT]-[TOPIC]-[date].html; the sidebar jumps to each phase. Drafted from ba-output/scope.md + workflow-register.md when a workspace exists; real numbers only, unknowns flagged as placeholders.
